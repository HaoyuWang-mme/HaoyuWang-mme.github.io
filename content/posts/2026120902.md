## 高并发调用链路对比：3种方案的实测数据

我用一个统一的测试脚本，模拟100次并发请求、单次请求平均消耗600 Token，记录**全量响应时间**和**总花费**。来进行对比。

### 方案A：串行循环（最笨也最贵）

javascript
async function qwenBatchSerial(prompts) {
  const results = [];
  for (let prompt of prompts) {
    const res = await callQwen(prompt); // 等上一个跑完才跑下一个
    results.push(res);
  }
  return results;
}


**实测数据**  
- 耗时：大约 78 秒  
- 花费：按[千聚api聚合平台](https://www.qianjuai.com/)qwen-max模型当前报价(约0.04元/1K token)→100次×600Token×0.04 = 2.4元（但这是理想值，实际因重试换行多出来10%左右）  
- 缺点：慢、无法利用并行、容易端口堆积引发超时重试（重试1次成本翻倍）

### 方案B：无限制并发（快速，但也烧钱）

javascript
async function qwenBatchThunder(prompts) {
  // 简单Promise.all，不限制并发数
  return Promise.all(prompts.map(p => callQwen(p)));
}


**实测数据**  
- 耗时：约 2.5 秒（因为100个请求几乎是同时发）  
- 花费：约 3.8 元（由于高并发中间触发了官方限流机制的退避，部分请求走了更高成本的重试链路，导致Token消耗暴增）  
- 缺点：瞬间打满API的速率限制，导致大量HTTP 429错误，重试浪费成本

### 方案C：限制并发数的任务队列（最优省钱方案）

javascript
// p-limit 库实现：可控制最大并发量为5
const pLimit = require('p-limit');
const limit = pLimit(5);

async function qwenBatchCool(prompts) {
  const tasks = prompts.map(p => limit(() => callQwen(p)));
  return Promise.all(tasks);
}


**实测数据**  
- 耗时：约 6.5 秒（虽然比2.5秒慢，但用户可接受）  
- 花费：约 2.7 元（比方案B省30%，比方案A省10%且快了10倍以上）  
- 结论：这是高并发场景下**成本和速度之间的甜点**。
