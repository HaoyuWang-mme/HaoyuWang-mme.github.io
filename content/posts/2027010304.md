## 第二节：高频调用优化核心——少花冤枉钱

Mistral API 按 token 计费，但实际消耗往往远超你的预期。以下 5 个秘籍能帮你至少省 30% 的费用。

### 秘籍 1：启用流式输出（stream=True）

很多新手习惯等完整回复再一次性拿到结果，但这样每次请求都会等待服务器生成完整内容，不仅时间长，还占用大量上下文缓存。使用流式输出可以边生成边消费，及时终止不必要的生成。

python
stream = client.chat.completions.create(
    model="mistral-small-latest",
    messages=[{"role": "user", "content": "讲一个笑话"}],
    stream=True
)
for chunk in stream:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="")


流式模式下，如果用户中途打断，可以立即停止请求，避免浪费后半段的 token。

### 秘籍 2：控制 max_tokens 和 stop 参数

很多人不设 `max_tokens`，结果模型输出到 4096 token 才停——实际用户可能只需要两句回复。设一个合理的上限，比如 500，能大幅减少浪费。

python
response = client.chat.completions.create(
    model="mistral-medium-latest",
    messages=[{"role": "user", "content": "简短介绍一下量子计算"}],
    max_tokens=300,
    stop=["。", "\n\n"]  # 在句号或连续换行处停止
)


同时利用 `stop` 参数指定终止序列，避免模型“唠嗑”。实测一个客服对话场景，设置 `stop=["用户：", "\n"]` 后，每轮对话 token 消耗减少 40%。

### 秘籍 4：请求缓存与去重

高频调用中，很多请求内容完全重复（比如用户问同一个常见问题）。在客户端加一层本地缓存，用消息内容的 hash 做 key，命中后直接返回结果，0 消耗。

python
import hashlib
import json
from functools import lru_cache

@lru_cache(maxsize=128)
def get_cached_response(messages_json: str):
    messages = json.loads(messages_json)
    response = client.chat.completions.create(
        model="mistral-small-latest",
        messages=messages,
        max_tokens=100
    )
    return response.choices[0].message.content

# 调用时序列化消息列表
user_input = [{"role": "user", "content": "今天天气怎么样"}]
reply = get_cached_response(json.dumps(user_input, ensure_ascii=False))


对于实时性不高的场景（如知识库查询），缓存命中率能到 60% 以上。

## 第四节：还有哪些容易忽视的坑
