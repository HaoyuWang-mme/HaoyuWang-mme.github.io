## 单点调用的三个致命陷阱

### 陷阱一：**接口崩了，业务就崩了**

这是最直接也最痛的一点。你辛辛苦苦把业务代码和某个 API 深度绑定，结果那个 API 宕机了，你的应用也直接断气。不用多，一次夜间故障，就能让你这个月的奖金和大佬的信任一起消失。

### 陷阱二：**计费膨胀，预算爆炸**

很多模型看起来便宜，但深入用起来，计费规则复杂得像迷宫。Token 消耗、并发限制、多轮对话的隐藏成本……你很难提前算清楚。最后拿了数据一看，好家伙，单月成本超出了当初预想的 3 倍。更恐怖的是，如果只依赖一个模型，你连对比和砍价的本钱都没有。

### 陷阱三：**锁定成本高，切换如换车**

代码里到处是 `model: "deepseek-chat"` 这种硬编码。等你想切到 Gemini 或者 Claude 了，好嘛，整个接口逻辑要重写，参数格式要调整，SDK 不兼容，还得重新调优 prompt。这中间的学习成本和开发时间，够你喝好几壶咖啡了。

这些陷阱加起来，本质就是一个问题：**开发者不该做应用的服务商保姆**。你应该专注于业务逻辑，而不是被特定的 API 绑定。
